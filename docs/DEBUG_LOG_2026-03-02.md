## 1. oxidation_state: 0 导致 "C0+" bug

### 问题
前端 `add_atom` / `add_atoms` 创建 species 时设置 `oxidation_state: 0`：
```typescript
species: [{ element: "C", occu: 1, oxidation_state: 0 }]
```

其 `species_string` 属性为 `"C0+"`。XYZ/CIF 输出中元素名变成 `C0+`。

### 修复

**前端 (`src/lib/structure/atom-manipulation.ts`)**
- `add_atom` (line ~262): 移除 `oxidation_state: 0`
- `add_atoms` (line ~298): 移除 `oxidation_state: 0`

**前端 (`src/lib/structure/index.ts`)**
- `Species` 类型: `oxidation_state: number` → `oxidation_state?: number`（改为可选）

**后端 (`server/routers/workflow.py`)**
- 改用 ASE 序列化（通过 `converter.py` 的 `_clean_element_symbol` 自动清理 'C0+' → 'C'）
- 即使前端仍有 `oxidation_state: 0`，后端也能正确处理

## 7. .gz 文件加载修复 (vite-plugin-gz-raw)

### 问题
侧边栏内置示例中 `.yaml.gz` 文件无法打开。YAML parser 报 `null byte is not allowed in input`。

### 根因
`desktop/Sidebar.svelte` 用 `import.meta.glob('$site/structures/*', { query: '?raw' })` 导入所有结构文件。
Vite `?raw` 将 `.gz` 二进制文件当 UTF-8 文本读取 → gzip 二进制被截断/乱码 → parser 收到含 null bytes 的字符串。

### 尝试过的方案

1. **`?url` + fetch + DecompressionStream** — 失败。`fetch(url)` 报 `Failed to fetch`，Vite dev server 不正确 serve `.gz` 文件的 URL。
2. **Vite 插件 `vite-plugin-gz-raw`** — 第一版 bug：`load(id)` 中先检查 `id.endsWith('.gz')`，但 Vite 传入的 `id` 带 `?raw` 后缀（如 `file.yaml.gz?raw`），导致判断失败，插件不生效。

### 最终修复 (`vite.desktop.config.ts`)

新增 `vite-plugin-gz-raw` 插件，在 Vite load 阶段用 Node.js `gunzipSync` 解压非 `.json.gz` 的 `.gz` 文件：

```typescript
{
  name: 'vite-plugin-gz-raw',
  enforce: 'pre',
  load(id) {
    const clean = id.replace(/\?.*$/, '')  // 关键：先去掉 ?raw/?url 后缀
    if (!clean.endsWith('.gz') || clean.endsWith('.json.gz')) return null
    const text = gunzipSync(readFileSync(clean)).toString('utf-8')
    return { code: `export default ${JSON.stringify(text)}`, map: null }
  },
}
```

**关键教训**: Vite `load(id)` 的 `id` 参数包含 query string（`?raw`, `?url` 等），必须先 `id.replace(/\?.*$/, '')` 再检查文件扩展名。

### 同时修改 (`desktop/Sidebar.svelte`)
- `handle_local_click` 中 `.gz` 后缀自动去除：`file.name.replace(/\.gz$/i, '')`，确保 parser 按正确格式识别文件
